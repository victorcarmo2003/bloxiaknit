## BLOXIA Runtime

This project is becoming the foundation runtime for BLOXIA. `Bloxia.Framework` owns service, controller, component and system lifecycle without a Knit backend. Lync owns gameplay networking through explicit shared schemas, and Jecs powers data-oriented simulation through `Bloxia.ECS`.

- Shared library entrypoint: `src/shared/Bloxia/init.luau`
- Framework entrypoint: `src/shared/Bloxia/Framework`
- Network source of truth: `src/shared/Bloxia/Net`
- Server services: `src/server/Services`
- Client controllers: `src/client/Controllers`
- Components: `src/server/Components` and `src/client/Components`
- Systems: `src/server/Systems` and `src/client/Systems`
- ECS registry: `src/shared/Bloxia/ECS`
- AI/project guidance: `metadata.json`
- Runtime/project config: `bloxia.config.json`
- Local CLI: `tools/bloxia.ps1`
- Framework guide: `.bloxia/framework.md`

Do not import Knit in gameplay modules. Use `Bloxia.Service`, `Bloxia.Controller`, `Bloxia.Component`, `Bloxia.System`, `Bloxia.ECS` and `Bloxia.Framework`. Components are tag-based instance modules and should live only under `src/server/Components` or `src/client/Components`. Systems are Jecs update loops and should live only under `src/server/Systems` or `src/client/Systems`. Add packets, queries and groups under `src/shared/Bloxia/Net/Schemas` and register schema modules in `Registry.luau` before `Bloxia.Net.start()`.

## AI Decision Map

- Needs server authority, public feature API, persistence or anti-cheat: create a `Service`.
- Needs client input, camera, local prediction, UI orchestration or presentation: create a `Controller`.
- Needs behavior attached to a tagged Roblox `Instance`: create a `Component`.
- Needs frame-based simulation over many entities/data rows: create a `System` plus ECS components.
- Needs client/server communication: create a `NetSchema`, register it, then consume it from modules.
- Needs reusable pure code without lifecycle: create a shared utility or feature module.

## BLOXIA CLI

Validate the template:

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File tools/bloxia.ps1 validate
```

Create modules from templates:

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File tools/bloxia.ps1 make service Inventory
powershell -NoProfile -ExecutionPolicy Bypass -File tools/bloxia.ps1 make controller Inventory
powershell -NoProfile -ExecutionPolicy Bypass -File tools/bloxia.ps1 make net Inventory
powershell -NoProfile -ExecutionPolicy Bypass -File tools/bloxia.ps1 make ecs Health
powershell -NoProfile -ExecutionPolicy Bypass -File tools/bloxia.ps1 make component Door --client
powershell -NoProfile -ExecutionPolicy Bypass -File tools/bloxia.ps1 make component Pickup --server
powershell -NoProfile -ExecutionPolicy Bypass -File tools/bloxia.ps1 make system Movement --server
powershell -NoProfile -ExecutionPolicy Bypass -File tools/bloxia.ps1 make system CameraSway --client
powershell -NoProfile -ExecutionPolicy Bypass -File tools/bloxia.ps1 make ui InventoryPanel
powershell -NoProfile -ExecutionPolicy Bypass -File tools/bloxia.ps1 make data Inventory
```

Cloud registry commands are prepared for Supabase and are intended to be wrapped by the BLOXIA IDE:

```powershell
$env:BLOXIA_SUPABASE_URL="https://your-project.supabase.co"
$env:BLOXIA_SUPABASE_ANON_KEY="your-anon-key"
powershell -NoProfile -ExecutionPolicy Bypass -File tools/bloxia.ps1 cloud search "flight system"
powershell -NoProfile -ExecutionPolicy Bypass -File tools/bloxia.ps1 cloud install bloxia.flight.basic --dry-run
```

The native IDE installer should verify artifacts, ask for required config, write files, register NetSchemas, then run `validate`.

## Getting Started
To build the place from scratch, use:

```bash
wally install
rojo build -o "BLOXIAFramework.rbxlx"
```

Next, open `BLOXIAFramework.rbxlx` in Roblox Studio and start the Rojo server:

```bash
rojo serve
```

For more help, check out [the Rojo documentation](https://rojo.space/docs).
